# 紧凑流式过程输出设计

## 目标

将助手执行任务时的流式过程统一呈现为“关键逻辑摘要 → 具体工具或命令”的时间线，帮助用户快速理解当前进展。过程输出不展示完整的模型中间叙述、内部任务原文或无关细节；最终回答仍在独立的消息区域完整呈现。

摘要只属于 UI 展示层投影。`ChatMessage.content`、`ChatMessage.blocks`、turn details 和会话持久化数据继续保留完整原始内容；展示层不得截断、改写或删除这些数据。

## 交互与数据流

1. 模型流式文本作为过程摘要候选进入渲染层。
2. 模型通过系统提示词主动将每条进度控制为最多五行的简短摘要；渲染层按原样展示，不做硬性截断或静默丢弃内容。
3. 工具调用继续使用现有结构化数据渲染，显示工具名称、目标路径或实际命令，不把命令输出混入摘要。
4. 摘要与后续工具按原有 block 顺序排列；当前正在执行的工具保持运行状态样式。
5. 流式结束后，过程时间线保留为可回顾的紧凑轨迹，最终回复在主内容区域显示一次，避免重复。

## 约束

- 摘要由模型控制在最多五行；前端不截断、不改写模型输出。
- 不新增 AI 摘要请求，避免额外延迟、费用和 IPC 链路。
- 工具命令、文件路径、错误状态等可操作信息不隐藏。
- 保持现有 thinking 显示设置和工具详情展开能力；本次只压缩面向用户的自然语言过程文本。
- 兼容中英文界面，不依赖固定文案判断流程。

## 实现边界

- 在 agent 系统提示中明确进度摘要格式和五行限制，并添加提示词测试，确保要求覆盖中英文界面。
- 调整流式过程文本的展示样式，使其明确区别于最终回答，但不改变消息存储和工具执行协议。

## 验证

- 运行新增及相关 agent/renderer 测试，确认进度提示词和展示行为。
- 运行 `bun run typecheck` 和 `bun run lint`。
- 通过现有开发服务器进行一次视觉检查，确认过程顺序为摘要、工具/命令、摘要、工具/命令，最终回答不重复出现在过程区。
