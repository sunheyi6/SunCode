# 聊天工具操作明细设计

## 目标

在聊天消息中完整展示编码代理执行过的文件编辑与命令操作，让用户可以直接确认：

- 操作了哪个文件；
- 文件当前处于 `编辑中`、`已编辑` 或 `编辑失败`；
- 本次操作增加和删除了多少行；
- 执行了什么命令；
- 命令的工作目录、标准输出、错误输出和退出码。

命令明细默认折叠，减少聊天区域的视觉噪音。

## 范围

本功能覆盖 SunCode 内置的 `edit`、`write` 和 `bash` 工具。

- `edit`：显示被修改文件及真实增删行数。
- `write`：新建文件显示全部新增行；覆盖文件显示覆盖前后的真实增删行数。
- `bash`：显示完整命令、工作目录、输出和退出码。

本期不推断 `bash` 命令间接造成的文件变化。通过命令执行前后比较整个工作区容易把用户已有的未提交修改错误归因给代理，因此该能力留待以后通过更明确的执行快照机制实现。

## 方案选择

采用结构化工具结果方案。工具层直接生成可供界面消费的元数据，而不是让渲染层解析人类可读的输出文本。

与其他方案相比：

- 解析输出文本改动较小，但输出文案变化会破坏解析。
- 每次命令前后执行 Git diff 能覆盖间接文件变化，但会混入用户原有修改，也无法可靠支持非 Git 工作区。
- 结构化结果需要扩展共享类型和消息存储，但数据准确、职责清晰，并且适合历史会话恢复。

## 数据模型

扩展 `ToolResult`，增加可选的结构化 `details` 字段。该字段使用带判别属性的联合类型：

```ts
type ToolResultDetails = FileEditDetails | CommandDetails;

interface FileEditDetails {
  type: 'file_edit';
  filePath: string;
  status: 'editing' | 'edited' | 'failed';
  addedLines?: number;
  removedLines?: number;
  error?: string;
}

interface CommandDetails {
  type: 'command';
  command: string;
  cwd: string;
  exitCode: number | null;
  signal?: string;
  stdout: string;
  stderr: string;
}
```

聊天消息中的每个 `ToolCallContent` 增加可选执行信息，包括状态和最终结果。工具调用开始时，聊天 store 将对应调用标记为运行中；工具结束时，将 `ToolResult` 写回同一个工具调用。

最终保存的 assistant 消息包含工具参数与工具结果，因此历史会话重新载入后仍能还原完整卡片。

## 文件增删行统计

文件工具在写入前保留旧内容，写入后比较新旧文本。

统计采用逐行差异算法，输出本次操作的新增行数和删除行数：

- 相同的行不计入变化；
- 新内容独有的行计入 `addedLines`；
- 旧内容独有的行计入 `removedLines`；
- 修改一行按删除一行并新增一行计算；
- 新建文件的 `removedLines` 为 `0`；
- 空文件按 `0` 行处理，不把空字符串误算为一行。

为避免 `edit` 和 `write` 各自实现一套算法，新增独立的纯函数模块负责行数统计。该模块不访问文件系统，可以直接进行行为测试。

## 工具执行数据流

### 文件编辑

1. 模型产生 `edit` 或 `write` 工具调用，参数中包含文件路径。
2. renderer 收到工具调用参数后创建文件操作卡片，状态显示 `编辑中`。
3. 工具读取旧内容并执行写入。
4. 工具计算旧内容与新内容的逐行差异。
5. 成功时返回 `file_edit` 结构，状态为 `edited`，包含增删行数。
6. 失败时返回 `file_edit` 结构，状态为 `failed`，包含文件路径和错误原因。
7. renderer 将结果更新到原卡片，并把中文状态映射为 `已编辑` 或 `编辑失败`。

如果调用参数尚在流式传输，卡片可以先显示工具名称；一旦参数形成有效 JSON，立即显示文件路径。

### 命令执行

1. 模型产生 `bash` 工具调用。
2. renderer 创建默认折叠的命令卡片。
3. 卡片标题优先使用工具参数中的 `description`，缺失时使用单行命令摘要。
4. 工具执行后返回 `command` 结构。
5. 展开卡片后显示完整命令、工作目录、标准输出、错误输出、退出码及可选信号。

输出仍保留现有截断上限，避免超大命令输出拖垮聊天渲染。

## 界面设计

新增统一的工具操作列表，放在对应 assistant 消息正文之后，并按实际调用顺序展示。

### 文件操作卡片

文件卡片默认展开且保持紧凑，包含：

- 文件图标；
- 项目内相对路径，无法转换时显示规范化绝对路径；
- 状态标签：
  - `编辑中`：使用进行中色；
  - `已编辑`：使用成功色；
  - `编辑失败`：使用错误色；
- 成功后的行数统计：绿色 `+N` 和红色 `-N`；
- 失败后的简短错误原因。

每次工具调用单独显示，不按文件合并。这样同一个文件被连续修改多次时，用户仍能看到执行顺序和每一步的增删量。

### 命令卡片

命令卡片使用原生或等价的折叠交互，默认关闭。

折叠标题包含：

- 终端图标；
- 命令描述或摘要；
- 执行中、成功或失败状态；
- 执行结束后的退出码。

展开内容包含：

- 完整命令；
- 工作目录；
- 标准输出；
- 错误输出；
- 退出码；
- 终止信号（存在时）。

无输出时明确显示“无输出”，而不是留下空白区域。

## 会话持久化

当前 assistant 消息在文本结束时过早保存，工具调用和结果可能尚未完成。调整为在整个 agent 回复完成后保存一次完整 assistant 消息。

保存内容包括：

- 正文；
- 思考内容（如有）；
- 工具调用参数；
- 工具执行状态和结构化结果。

加载历史消息时，chat store 直接恢复工具调用数组。旧会话没有结构化结果时继续兼容：仍可显示工具名与参数，但不伪造文件统计或命令退出信息。

## 错误处理

- 文件路径参数缺失或 JSON 尚未完整时，显示工具名称，等待后续参数或结果补齐。
- 文件操作失败时，工具尽可能返回已解析的目标路径和失败原因。
- 命令无法启动时，显示失败状态、命令本身和启动错误；退出码为 `null`。
- 未识别的工具继续使用通用工具卡片，不阻塞其他工具记录。
- renderer 不从人类可读 `output` 中推断结构化字段；缺少结构化信息时采用兼容展示。

## 组件边界

- 共享类型：定义文件编辑和命令结果结构。
- 行差异工具模块：只负责旧文本与新文本的增删行计算。
- `edit` / `write` 工具：负责采集旧内容、写入文件并生成结构化结果。
- `bash` 工具：负责生成结构化命令结果。
- chat store：将工具开始、参数流和工具结束事件绑定到当前 assistant 消息，并负责完整消息持久化。
- 工具展示组件：根据结构化结果选择文件卡片、命令卡片或通用回退卡片。
- assistant 消息组件：只负责将当前消息的工具调用交给工具展示组件。

## 测试与验证

项目当前没有通用单元测试脚本。实施时为纯逻辑引入轻量、可由 Bun 直接运行的测试文件，不增加大型测试框架。

至少验证：

- 新建文件的新增行数；
- 删除、增加和修改行的统计；
- 空文件与末尾换行；
- `edit` 成功、失败的结构化结果；
- `write` 新建和覆盖的结构化结果；
- 命令成功、非零退出和启动失败的结构化结果；
- chat store 将工具结果绑定到正确调用；
- 历史消息保存和恢复后仍包含操作明细；
- 命令卡片默认折叠；
- 旧消息缺少结构化结果时能够兼容渲染。

完成后运行：

```text
bun run typecheck
bun run lint
bun run build
```

并手动执行一次文件编辑、一次失败编辑、一次成功命令和一次失败命令，确认聊天展示和历史恢复行为。
